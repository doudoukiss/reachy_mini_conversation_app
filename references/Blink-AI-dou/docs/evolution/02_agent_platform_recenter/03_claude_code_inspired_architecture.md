# 03 — Claude-Code-inspired intelligence architecture

The goal is not to copy a coding assistant literally.
The goal is to borrow the deepest ideas that make Claude Code structurally strong.

## The key ideas to borrow

### 1. Persistent instruction memory
Claude Code uses project memory files to keep long-lived operating instructions stable across sessions.

Blink-AI should adopt the same idea with explicit runtime instruction layers:

- `IDENTITY.md`
  - who Blink-AI is
  - tone and social posture
  - honesty and embodiment limits
  - rules about not inventing what it sees or knows

- `SITE_POLICY.md`
  - current venue rules
  - opening/closing behavior
  - accessibility notes
  - escalation rules
  - when to be proactive vs quiet

- `BODY_POLICY.md`
  - allowed expressions
  - gaze constraints
  - no-motion / safe-idle behavior
  - expression rules under uncertainty

- `USER_MEMORY.md`
  - user preferences and standing facts
  - should be generated by the system, not hand-edited casually

This is better than burying everything inside prompt strings.

## 2. Skills as explicit capability packs
Claude Code skills are markdown playbooks for specific jobs.

Blink-AI should have skills such as:

- `welcome_guest`
- `wayfinding`
- `calendar_lookup`
- `general_conversation`
- `observe_and_comment`
- `remember_and_follow_up`
- `incident_escalation`
- `safe_degraded_response`
- `investor_demo_mode`

Each skill should define:
- when it should trigger
- required tools
- forbidden claims
- expected output structure
- success criteria

This keeps intelligence inspectable.

## 3. Subagents with bounded responsibilities
Claude Code subagents separate concerns and preserve context.

Blink-AI should introduce subagents like:

### Perception analyst
Input:
- current frame(s)
- recent world state
- recent speaker metadata

Output:
- structured scene facts only
- no freeform persona chatter

### Dialogue planner
Input:
- transcript
- world model
- retrieved memory
- venue facts
- active skill

Output:
- response plan
- suggested tools
- candidate action intention

### Safety / policy reviewer
Input:
- candidate reply
- candidate action plan
- current runtime constraints

Output:
- approved / corrected / downgraded reply and action plan

### Memory curator
Input:
- turn transcript
- scene facts
- outcome

Output:
- short-term memory updates
- semantic memory candidates
- user profile updates
- episode annotations

### Reflection / eval agent
Input:
- completed turn or scene
- traces
- latency
- outcome

Output:
- scorecard
- error tags
- proposed improvement notes

Do not let one giant agent perform all of these roles.

## 4. Hooks at lifecycle boundaries
Claude Code hooks fire at key lifecycle moments.

Blink-AI should implement deterministic hook surfaces:

- `before_listen`
- `after_transcript`
- `after_perception`
- `before_reply_generation`
- `before_speak`
- `after_turn`
- `on_safe_idle`
- `on_escalation`
- `on_provider_failure`
- `on_shift_tick`

These hooks can:
- run rules
- call lightweight subagents
- update logs
- produce audit events
- block unsafe action plans

## 5. Strict tool schemas
A robotics intelligence system needs strict, typed tools.

Blink-AI should expose all intelligence capabilities through a single tool registry with strict JSON schemas.

Tool categories:

### Knowledge tools
- venue FAQ lookup
- calendar lookup
- room lookup
- staff lookup
- imported document search

### Perception tools
- latest scene facts
- recent participant list
- visible text
- current engagement estimate
- current attention target

### Memory tools
- search semantic memory
- fetch recent episode summaries
- read user profile
- write memory candidate
- mark memory rejected

### Action tools
- set expression
- set gaze
- perform gesture
- perform animation
- safe idle
- later: write serial body frame

### System tools
- get runtime status
- get current model profile
- get latency snapshot
- provider health
- storage usage
- battery / network state when available

The planner should never write raw servo positions directly.

## 6. Status line and operator transparency
Claude Code’s status surfaces matter.

Blink-AI should display:
- active skill
- current subagent decisions
- current model backend
- voice backend
- perception freshness
- confidence level
- fallback state
- safe-idle state
- body mode
- current memory retrieval sources

This increases investor trust because the system looks like an operating runtime, not a black box toy.

## 7. Session compression and memory discipline
Claude Code’s context discipline is a major lesson.

Blink-AI should avoid endlessly appending raw transcript.
Instead it should maintain:

- raw conversation log
- short rolling summary
- structured memory objects
- compact user profile
- explicit scene history
- explicit unresolved threads

This is especially important on local hardware.

## Resulting architectural slogan

### Blink-AI should become:
**an embodied agent operating system**

not:
**a chatbot with robot commands**

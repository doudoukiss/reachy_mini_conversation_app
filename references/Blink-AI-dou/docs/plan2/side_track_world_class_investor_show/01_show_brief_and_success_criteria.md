# Show Brief And Success Criteria

## Objective

Build a **ten-minute investor performance** that makes Blink-AI feel:

1. alive,
2. grounded,
3. commercially believable,
4. safe,
5. technically real,
6. and ready for disciplined scale.

The performance should be strong enough to use when live demo conditions are degraded by:
- network instability,
- model-provider outages,
- camera unreliability,
- room noise,
- or timing pressure.

## Product-aligned framing

The performance must reinforce the repo's current direction, not blur it.

The audience should leave with these conclusions:

- Blink-AI is a **local-first companion**, not just a robot shell.
- The robot head is an **optional projection** of the same companion runtime.
- The system is **grounded, inspectable, and replayable**.
- The team already understands **operator oversight and safe fallback**.
- The commercial wedge is believable in **community concierge / guide deployments**.
- The path from today's demo to future deployment is an **engineering expansion**, not a rewrite.

## Use only what is already believable in the repo

The show should primarily use existing implemented capabilities already evident in the repository:

- semantic head embodiment over the serial head path
- virtual / inspectable presence shell
- investor-scene runner and curated scene catalog
- perception fixtures and manual/perception snapshot paths
- venue knowledge and deterministic help flows
- memory capture and follow-up recall
- operator escalation and incident timeline visibility
- safe-idle behavior on degraded transport / low battery
- session / run / episode export and evidence bundles
- console visibility into state, traces, and outcomes

Thin new scaffolding is acceptable only when it serves orchestration, polish, or reliability.

## The world-class bar

A world-class investor show is not a kitchen-sink feature dump.

It has the following properties:

### 1. Every minute delivers one clear investor inference
The audience should never wonder why a segment matters.

### 2. Every claim has a proof beat
If the robot says something important, the screen or motion should prove it.

### 3. No dead air
Nothing should feel like waiting for a hidden process.

### 4. Physical motion is purposeful
The head should feel expressive and composed, not twitchy, cute, or unsafe.

### 5. Safety is part of the show
The fallback path should increase confidence, not feel like a failure.

### 6. The demo is honest
No silent wizard-of-oz behaviors. No hidden teleoperation masquerading as autonomy.

### 7. The close lands the product thesis
The ending should explain why this is a real company and not a one-off robot trick.

## Core demo design thesis

The best investor performance here is:

**A scripted keynote delivered by the companion itself, while the runtime quietly proves each chapter underneath.**

That is stronger than either extreme:
- stronger than a pure canned puppet show
- stronger than a fully live, fragile, improv-heavy demo

The right structure is:

### Presence beat
The robot immediately feels like one coherent presence.

### Proof beat
A real runtime action occurs and leaves inspectable state.

### Business beat
The audience understands why the capability matters commercially.

Repeat that rhythm across the ten minutes.

## Recommended audience experience

### Visual surfaces
Use three surfaces:

1. **Hero surface**
   - the physical robot head
2. **Presentation surface**
   - a clean show page or curated presence shell on the projector
3. **Operator proof surface**
   - the console on a side monitor, not the main screen

### Audio strategy
The spoken demo should come from a local path:
- `macos_say` first
- optional pre-rendered local audio as a later polish layer

### Input strategy
Avoid requiring live audience input during the ten-minute performance.
Use:
- prewritten prompts
- perception fixtures
- scripted scene runs
- deterministic state injection

Open Q&A can happen after the ten-minute show ends.

## Narrative arc

The arc should be:

1. **I am here**
   - presence, orientation, confidence
2. **I am one mind across surfaces**
   - same companion, optional embodiment
3. **I can understand what is in front of me**
   - grounded perception
4. **I can remember and stay useful**
   - memory continuity plus venue help
5. **I know when to hand off**
   - operator oversight and escalation
6. **I fail honestly and safely**
   - safe idle, explicit degraded behavior
7. **I leave evidence**
   - traces, artifacts, replayability
8. **I fit a credible product strategy**
   - local-first companion, optional embodiment, deployable wedge

## Segment selection discipline

The ten-minute show should prioritize these capabilities over everything else:

### Must-show
- presence / embodiment
- grounded perception
- memory continuity
- venue usefulness
- escalation / oversight
- safe fallback
- evidence export

### Optional insert, only if timing is strong
- two-person attention handoff
- disengagement shortening
- operator correction after wrong interpretation

These are valuable, but should not dilute the first-version story.

## Reliability strategy

The performance should be engineered to be boring in the best way:

- **prewritten spoken lines**
- **silent proof scene execution** underneath the speech
- **local TTS or pre-rendered local audio**
- **fixture-based perception**, not live dependence
- **explicit expected-output assertions** for proof cues
- **fallback narration lines** per segment
- **artifact export at the end of every run**

This keeps the show polished without pretending it is something it is not.

## Investor takeaways by chapter

### Chapter 1 — Presence
Takeaway: this feels alive, not static.

### Chapter 2 — Same mind across surfaces
Takeaway: embodiment is downstream of the core product, not the whole company.

### Chapter 3 — Grounding
Takeaway: the system can answer from current evidence, not vibes.

### Chapter 4 — Memory and usefulness
Takeaway: this can become a sticky deployed product.

### Chapter 5 — Oversight
Takeaway: the team understands real-world operation, not just autonomy theater.

### Chapter 6 — Safety
Takeaway: the company will not destroy trust in deployment.

### Chapter 7 — Evidence
Takeaway: the platform is measurable, repeatable, and investable.

## Hard non-goals

Do not turn the show into any of the following:

- a generic chatbot in a robot shell
- a cloud-only magic trick
- a raw-servo choreography experiment
- a toy-like personality showcase
- a hidden teleoperation demo
- a UI overload with too many panels
- a hardware-first product pitch that contradicts repo direction

## Definition of success

The show is successful when an investor can say, in one sentence:

> "This is a real local-first companion system with optional embodiment, grounded behavior, operator-safe deployment logic, and a believable path to product."

If the show lands that sentence, it is doing its job.

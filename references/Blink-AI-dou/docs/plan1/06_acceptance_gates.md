# Acceptance Gates

## Product thesis gates
- top-level docs all describe the same product
- terminal-first companion is the hero path
- concierge is explicitly a mode, not the baseline identity
- the body is clearly optional

## Runtime gates
- default daily mode is fluid enough for real conversation
- model profile names reflect product intent
- the router exposes latency and backend choices clearly
- local-only mode remains supported but is not forced as the flagship path

## Skill and prompt gates
- core identity prompts are personal companion first
- venue prompts only activate in venue mode
- skills are grouped around companion behavior, not demo history

## User experience gates
- a user can start Blink-AI and talk in terminal without reading a long setup doc
- the browser console is useful but optional
- the system feels persistent and relationship-aware across sessions

## Embodiment gates
- robot embodiment remains downstream of semantic state
- no brain-side raw servo logic leaks upward
- body absence never breaks the product

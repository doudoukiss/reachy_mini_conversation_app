# Companion Core Feature Map

## Goal

Define what makes Blink-AI feel like a companion rather than a demo stack.

## Capability groups

### 1. Conversation
- fluid ordinary dialogue
- interruption handling
- concise or expansive answer modes
- honest uncertainty
- follow-up coherence

### 2. Memory
- stable user profile memory
- episodic recall
- semantic recall
- unresolved thread tracking
- relationship timeline summaries

### 3. Observation
- notice who is present
- describe scene changes when invited or useful
- observe screen/desk context conservatively
- track freshness and confidence

### 4. Planning and support
- daily planning
- reminder follow-up
- bounded note-taking
- safe browser or local actions when approved

### 5. Relationship behavior
- greet appropriately
- reference prior context without overdoing it
- use memory to feel continuous, not creepy
- adjust style by context and user state

### 6. Embodiment projection
- optional semantic body actions
- voice tone / phrasing coherence
- robot head later maps from the same semantic state

## Companion modes

### `companion_live`
Default daily mode.
Fastest fluent experience.

### `companion_private_local`
Local-first personal mode.

### `focus_mode`
Less interruption, shorter answers, no unnecessary proactivity.

### `venue_demo`
Explicit demo / concierge mode.

## Not yet core product

Do not define the product around:
- generic browser automation breadth
- large robot movement catalogs
- community-site content alone
- investor-specific scripts

## Acceptance standard

If the robot body were absent for a year, Blink-AI should still feel like a real product.
If that sentence is not true, the architecture is still too robot-first.

# Community Applications

This document describes an explicit venue/demo vertical for Blink-AI. It is not the default identity of the product.

## Best early markets

- community centers
- maker spaces
- libraries
- museums
- campuses
- events and conferences
- elder-care common spaces
- youth programs

## Strong early tasks

- greeting and onboarding
- answering common questions
- wayfinding
- event guidance
- schedule lookup
- volunteer / staff handoff
- simple educational explanations
- collecting structured feedback

## Why this is strategically good

These settings reward:

- clear value from conversation and memory
- visible embodiment
- repeatable demos
- low initial hardware complexity
- strong software moat through venue data packs, operator tooling, and workflows

## What to avoid in the first pitch

- generic humanoid claims
- dexterous manipulation claims without proof
- open-world autonomy claims
- “it can do anything” positioning

## Better positioning

> A venue-mode deployment of Blink-AI: a local-first companion system adapted into a nearby-brain embodied community guide.

## Why venue intelligence matters

For early pilots, Blink-AI should not depend on hand-coded demo facts.

It becomes more commercially credible when it can ingest:

- a site FAQ pack
- a real event schedule
- room and signage data
- staff contact information
- operational markdown notes

That turns the venue-mode story from “seeded venue-guide demo” into “venue intelligence system with an embodied interaction layer,” without redefining the core product away from personal companion use.
